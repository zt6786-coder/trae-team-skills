# Summarize - 内容摘要

## 概述

本技能为多种内容类型提供结构化摘要服务，支持网页URL、PDF文档、音频转录文本、视频字幕等。根据内容类型和用户需求，生成不同粒度的摘要，保留关键信息和数据，标注来源。

## 支持的内容类型

| 内容类型 | 识别方式 | 内容提取方式 |
|----------|----------|-------------|
| 网页URL | 以http/https开头 | 使用defuddle技能或WebFetch提取干净内容 |
| PDF文档 | .pdf后缀或PDF文件 | 使用pdf技能提取文本 |
| 纯文本 | 直接粘贴的文本 | 直接处理 |
| YouTube视频 | youtube.com链接 | 获取字幕/转录文本 |
| 音频转录 | 转录文本 | 直接处理转录稿 |
| 长文档 | 用户直接提供的长文 | 按章节分块处理 |

## 摘要类型

### 1. 执行摘要（Executive Summary）
- **长度**：3句话
- **适用场景**：快速了解核心内容，给领导/同事转述
- **要求**：
  - 第1句：核心主题/结论
  - 第2句：关键论据/发现
  - 第3句：重要意义/建议/后续行动

### 2. 要点摘要（Bullet Points）
- **长度**：5-10个要点
- **适用场景**：快速扫读、做笔记
- **要求**：
  - 每个要点1-2句话
  - 按重要性排序
  - 包含关键数据和结论
  - 互不重叠

### 3. 详细摘要（Detailed Summary）
- **长度**：原文的10-20%
- **适用场景**：深入了解内容，替代阅读原文
- **要求**：
  - 按原文结构分章节
  - 保留核心论点和论据
  - 关键数据不遗漏
  - 标注重要引用的原文位置

### 4. 关键数据提取（Key Data Extraction）
- **格式**：结构化列表/表格
- **适用场景**：提取报告中的数字、统计、事实
- **要求**：
  - 所有数字+单位+上下文
  - 标注数据来源
  - 区分事实和预测/估计
  - 注意数据时效性

### 5. 行动项提取（Action Items）
- **格式**：待办事项列表
- **适用场景**：会议纪要、邮件、任务文档
- **要求**：
  - 明确谁做什么
  - 标注截止时间（如有）
  - 标注优先级
  - 区分已决定事项和待讨论事项

### 6. TL;DR（Too Long; Didn't Read）
- **长度**：1句话
- **适用场景**：社交媒体、聊天中快速分享
- **要求**：一句话说清楚最核心的信息

## 工作流程

### Step 1: 识别内容类型
```
用户输入URL/文件/文本
    ├─ URL → 判断是网页/视频/PDF → 提取内容
    ├─ 文件 → 根据后缀判断类型 → 提取内容
    └─ 文本 → 直接进入摘要
```

### Step 2: 提取干净内容
- **网页**：优先使用defuddle技能提取正文，去除导航/广告/侧栏
- **PDF**：使用pdf技能提取文本，保留章节结构
- **视频**：提取字幕/转录文本，去除时间戳冗余
- **长文本**：按章节/段落分块，确保上下文完整

内容提取质量检查：
- 是否去除了无关内容（导航、广告、版权声明）
- 是否保留了标题层级结构
- 关键数据和引用是否完整
- 特殊字符是否清理干净

### Step 3: 判断摘要类型
根据用户需求选择：
- 用户说"简单说一下" → TL;DR 或 执行摘要
- 用户说"要点"、"重点" → 要点摘要
- 用户说"详细总结"、"完整概括" → 详细摘要
- 用户说"数据"、"数字" → 关键数据提取
- 用户说"要做什么"、"action items" → 行动项提取
- 用户未指定 → 默认要点摘要

### Step 4: 生成摘要
按对应摘要类型的模板和质量标准生成。

### Step 5: 标注来源和元信息
在摘要末尾附加：
```
---
**来源**：[标题](URL)
**作者**：xxx（如可获取）
**发布日期**：xxxx-xx-xx（如可获取）
**摘要生成时间**：xxxx-xx-xx
```

## 各内容类型摘要模板

### 网页/文章摘要模板（要点摘要）
```
## 《{文章标题}》摘要

**核心观点**：（一句话核心结论）

### 关键要点
- **{要点1}**：（展开说明，含数据/论据）
- **{要点2}**：...
- **{要点3}**：...

### 重要数据/事实
- （数据1：数字+单位+含义）
- （数据2：...）

### 文章结论/建议
（作者的最终结论或行动建议）

---
**来源**：[{标题}]({URL})
```

```
**报告信息**

- 发布时间：xxxx-xx
- 页数：xx页

### 执行摘要
（3句话核心结论）

### 各章节要点
**第一章：{章节标题}**
- （要点1）
- （要点2）

**第二章：{章节标题}**
- ...

### 关键发现

2. ...

### 核心数据汇总
| 指标 | 数值 | 说明 |
|------|------|------|
| ... | ... | ... |

---
**来源**：{文件名/URL}
```

## 质量标准

### 好的摘要应该：
1. **准确性**：不歪曲原意，不添加原文没有的信息
2. **完整性**：关键信息不遗漏（核心论点、重要数据、结论）
3. **简洁性**：去除冗余、例子、铺垫，保留干货
4. **结构化**：使用标题、列表、表格组织信息
5. **客观性**：摘要本身不夹带个人观点（可加"备注/简评"段落标注个人看法）
6. **可溯源**：重要引用标注位置（章节/时间戳/段落）

### 常见错误要避免：
- 把引言当成结论（开头往往是铺垫，不是重点）
- 遗漏关键数据（数字比描述重要）
- 摘要长度不当（太短丢信息，太长不叫摘要）
- 混入个人推测（区分作者观点和自己的理解）
- 忽略反面观点（争议内容要呈现双方立场）

## 使用defuddle提取网页内容

当用户提供URL时，优先使用defuddle技能提取干净内容：

```
调用defuddle技能 → 获取clean markdown → 进行摘要
```

如果defuddle不可用，回退方案：
1. 使用WebFetch获取页面
2. 手动去除导航、广告、页脚等无关内容
3. 提取正文区域
4. 进行摘要

## 长文本处理策略

当内容超过模型上下文窗口时：
1. **分块处理**：按章节/段落分块，每块单独摘要
2. **递归摘要**：先对每块生成摘要，再对所有块摘要生成总摘要
3. **结构优先**：先提取目录/标题结构，再填充各部分要点
4. **重点保留**：优先保留结论、数据、建议，压缩背景和例子

## 语言适配

- 中文内容 → 中文摘要
- 英文内容 → 默认中文摘要（除非用户要求英文）
- 中英混合 → 中文摘要，保留英文专有名词
- 摘要语言与用户提问语言一致

## 注意事项

1. 摘要不等同于翻译，不要逐句翻译英文内容
2. 付费内容/登录墙内容可能无法提取，告知用户
3. 视频没有字幕时，告知用户无法自动获取内容
4. 涉及医疗/法律/投资的内容，摘要标注"仅供参考，不构成专业建议"
5. 非常长的内容（书籍、长报告）建议分章节多次摘要
